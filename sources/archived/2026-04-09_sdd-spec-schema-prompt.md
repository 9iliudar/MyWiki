本轮对话中，用户已经稳定理解以下四个概念及其关系：

1. SDD（Spec-Driven Development）
- 本质是先定义清楚，再开发。
- 重点不只是“写得更规范”，而是减少隐性分歧。
- 让开发有方向、有目标、有范围。

2. Spec
- Spec 是 Specification 的简称。
- 用户明确理解：Spec 是把需求转化为执行标准和验收依据。
- Spec 偏原则、偏整体约束、偏黑盒视角。
- 它回答的是：系统应该做什么，做到什么程度算对。

3. Schema
- Schema 是结构规则。
- 用户明确理解：Schema 约束对象内部应该长成什么样。
- Schema 偏结构、偏内部组织、偏白盒视角。
- 它回答的是：数据或内容应该长什么样，结构是否合法。

4. Prompt
- Prompt 不只是提示词，更准确地说，是给模型的即时操作指令。
- 它把某次具体任务中的 Spec 和 Schema 交给模型执行。
- Prompt 偏执行层。

核心金句：Spec 管原则，Schema 管结构，Prompt 管执行。

进一步理解：
- Spec 定义目标、边界、规则、验收标准。
- Schema 定义数据或内容的内部结构。
- Prompt 把前两者翻译成某次具体任务的执行指令。

关于四者关系的稳定结论：
- SDD 是方法框架。
- Spec、Schema、Prompt 是这个框架里最关键的三个工作层次。
- 可以理解为：SDD 用 Spec 驱动开发，用 Schema 固定结构，用 Prompt 驱动模型执行。
