# Wiki Schema

本文档定义 MyWiki 当前的正式内化规则。它不是“尽量多建页”的规则，而是“受控增长”的规则。

## 核心原则

- 精不求多，消化一个，进一个。
- 只有用户已经理解或明确掌握的概念，才能进入正式 wiki。
- 只是提到、顺带解释或尚未掌握的概念，进入候选层或停留在原始素材中。
- 优先更新已有页面，避免重复建页。
- 文件夹负责主分类，双链和标签负责跨领域关系。

## 目录结构

正式知识：

- `wiki/MyWiki/<Category>/`

候选概念：

- `wiki/candidates/<Category>/`

原始素材：

- `sources/inbox/`
- `sources/archived/`

Web 发布产物：

- `web/pages/`
- `web/data/`

## 分类规则

- 每个正式页面只放一个主分类目录。
- 当前默认分类是 `AI`。
- 跨领域概念不要复制到多个目录，改用 `[[双链]]` 和标签表达。

## 页面 Frontmatter 规范

每个正式页面必须包含以下字段：

```yaml
---
title: 页面标题
category: 所属分类
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - 来源素材路径
related:
  - "[[相关页面]]"
tags:
  - 标签1
  - 标签2
evolution:
  - "YYYY-MM-DD: 变更说明"
---
```

## 标题与术语规范

- 页面正文统一用中文表达为主。
- 技术术语保留英文原文。
- 如果页面涉及英文术语，术语在标题或正文首次出现时补上国际音标。
- 推荐格式：
  - `Prompt /prɑːmpt/`
  - `Schema /ˈskiːmə/`
  - `RAG /ræɡ/（检索增强生成）`
- 文件名 `name` 保持简洁稳定，不带音标。
- `title` 可以包含音标。

## 正式页与候选页规则

### 正式页

只有满足以下条件，才应进入正式 wiki：

- 用户已经理解该概念。
- 该概念是本轮对话的核心收获。
- 页面内容足够完整，不是几句浅层解释。

### 候选页

满足以下情况时，应进入候选层而不是正式 wiki：

- 用户只是初步了解。
- 概念只是辅助说明，不是本轮主目标。
- 当前还不适合独立成页。

## Ingest 规则

每次 ingest 应遵循以下约束：

1. 先输出 3-5 句摘要。
2. 判断用户真正掌握了哪些概念。
3. 默认最多只自动内化 `1-2` 个正式概念。
4. 其余概念进入 `candidate_concepts`，不要强行建正式页。
5. 如果某个点只是父概念的一部分，优先写入父页，不要拆成新页。
6. 正式页面正文必须完整、不断句、不少于 180 字。
7. 更新 `wiki/index.md` 和 `wiki/log.md`。

## 页面内容规范

每个正式页面应尽量包含：

- 简明定义
- 核心要点
- 与其他概念的关系
- 一句压缩结论

如果是英文术语页，建议在开头定义中直接给出：

- 英文术语
- 国际音标
- 中文解释

## Lint 检查规则

巡检时重点关注：

1. 矛盾页面
2. 孤岛页面
3. 过浅页面
4. 重复页面
5. 缺失目标的双链
6. 明显应互相关联却未关联的页面

## Index 规则

`wiki/index.md` 不是全量索引表，而是正式知识的清晰导航。

- 按分类或主题分组
- 只收正式知识
- 优先保留稳定、高价值页面入口

## Log 规则

`wiki/log.md` 记录正式操作历史，例如：

- ingest
- query 回写
- lint 修复
- candidate 提升为正式页

建议格式：

```md
## YYYY-MM-DD HH:MM

**操作**: ingest / query / lint / promote
**素材**: 来源文件
**影响页面**: 页面A, 页面B
**摘要**: 本次变化的简短说明
```
